# RFC-v0.14-003 — Typed Catalog Structs and Service Cookbook

**Status:** Implemented (v0.14.0)
**Target version:** v0.14.0
**Parent:** v0.14-001.
**Cross-refs:** RFC v0.9-003 (semantic toolkit), v0.5-004 (catalog v1).

## 1. Problem

`fjell-semantic-toolkit` (RFC v0.9-003) shipped one hand-written typed
emitter (`emit_update_staging_advanced`) and deferred the rest "to
v0.9.1". v0.14 picks up that deferred work as part of validating
the SDK author experience: untyped `encode(tag, fields)` calls are
exactly the kind of surface that produces silent runtime errors.

In parallel, v0.14 needs a cookbook of small worked examples — "how
to do X with the SDK" — so future authors have idiomatic patterns to
start from, not blank pages.

## 2. Generated typed emitters

For every entry in `CATALOG_V1`, the toolkit auto-generates:

```rust
pub struct <Name>Args { /* one field per IntentSchema FieldDef */ }

pub fn emit_<snake_name>(
    args: &<Name>Args,
    tick: u64,
    out: &mut [u8],
) -> Result<usize, SemanticError>;
```

For example, `UPDATE.STAGING_FAILED` (0x0102) becomes:

```rust
pub struct UpdateStagingFailedArgs {
    pub candidate_id: u32,
    pub error_code:   u16,
}

pub fn emit_update_staging_failed(
    args: &UpdateStagingFailedArgs,
    tick: u64,
    out: &mut [u8],
) -> Result<usize, SemanticError> { ... }
```

Generation rules:

- Field name → struct field name in snake_case.
- Field kind → Rust type (`U8`→`u8`, `Bytes32`→`[u8; 32]`).
- Required fields → struct fields without `Option`.
- Optional fields → `Option<T>` fields; `None` → `FieldValue::Absent`.

The generated code lives at `crates/fjell-semantic-toolkit/src/generated/`,
checked into the repo (not built fresh every compile — RFC v0.10-003
reproducibility constraint). Re-generated by `cargo xtask toolkit
regenerate` when the catalog changes.

## 3. Generation tooling

A new bin: `cargo xtask toolkit regenerate`:

- Reads `CATALOG_V1` directly.
- Renders one Rust module per catalog entry.
- Writes to the `generated/` tree.
- Adds the new generated symbols to `fjell-sdk`'s `sdk_emit` module
  via a `pub use` re-export.
- The generation is fully deterministic; running twice produces
  byte-identical output (verified by `repro-check`).

The generated symbols carry the same stability tier as the catalog
itself: STABLE per RFC-v0.10-002 §3.

## 4. Hand-authored documentation

For each generated emitter, a docstring block from the catalog entry
describing intent, required state, and typical caller. This makes
`cargo doc --no-deps -p fjell-sdk` immediately useful.

The toolkit also emits a per-domain table to `docs/sdk/intents.md`:

```text
## UPDATE domain (tags 0x0100–0x01FF)

| Tag | Symbol | Emitted by | When |
|-----|--------|-----------|------|
| 0x0101 | UPDATE.STAGING_ADVANCED | upgraded | candidate advances state |
| 0x0102 | UPDATE.STAGING_FAILED   | upgraded | candidate fails staging |
| ... |
```

## 5. Service author cookbook

`docs/sdk/cookbook.md` ships with worked examples derived from
`fjell-config-sync` (v0.14-002) and the existing services:

- **R1.** "How do I receive an IPC message and reply?"
- **R2.** "How do I emit a semantic intent?" (uses typed emitter from
  this RFC).
- **R3.** "How do I request a capability and check rights?"
- **R4.** "How do I open a persistent store and bind a lease?"
- **R5.** "How do I emit an audit record?"
- **R6.** "How do I declare what I need in my CapManifest?"
- **R7.** "How do I package and sign my bundle?"
- **R8.** "How do I write a service test using dev-harness?"

Each recipe is ≤ 80 lines, with code blocks marked `bash run-verified`
and `rust run-verified` so RFC-v0.10-006's truth-bond mechanism
catches drift.

## 6. Interaction with existing toolkit code

The hand-written `emit_update_staging_advanced` (RFC v0.9-003 §7)
remains for one release as a deprecation example; it logs a
deprecation warning to `cargo doc` and the recipe R2 calls the new
generated symbol. After v0.14, the hand-written version is removed in
v0.14.1.

The fixture generator from v0.9-003 §2 is now bound to the generated
emitters; fixture round-trip tests gain N entries automatically.

## 7. Acceptance criteria

1. `cargo xtask toolkit regenerate` runs and produces the
   `generated/` tree.
2. The generated tree is deterministic (verified by repro-check).
3. Every catalog v1 entry has a typed emitter + Args struct.
4. The typed emitters re-export through `fjell-sdk::sdk_emit::*`.
5. The fixture round-trip suite passes against every generated
   emitter.
6. `docs/sdk/intents.md` exists and covers every domain.
7. `docs/sdk/cookbook.md` exists with R1–R8 verified.
8. The reference service `fjell-config-sync` uses generated emitters
   (no `encode(tag, ...)` direct calls).
9. The hand-written `emit_update_staging_advanced` is deprecated with
   a docstring redirect; removal scheduled for v0.14.1.

## 8. Out of scope

- Generation for catalog v2 (catalog v2 does not exist).
- Procedural macros (`#[derive(IntentEmitter)]` etc.). The
  off-line generated approach is preferred for v1.0 because it keeps
  build determinism simple and surface inspectable.
- Multi-language bindings for the catalog. Rust only.
- Auto-generation of *consumers* (decoders into typed structs).
  Doable but defers to v1.x.
