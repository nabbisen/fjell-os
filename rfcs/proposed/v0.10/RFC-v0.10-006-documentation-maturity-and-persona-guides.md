# RFC-v0.10-006 — Documentation Maturity and Persona Guides

**Status:** Proposed
**Target version:** v0.10.0
**Parent:** RFC 061 (releasability).
**Affects:** `docs/src/`, `README.md`.

## 1. Problem

`docs/src/` exists as an mdbook skeleton. The project's design
guidelines list three personas (new users, intermediate users,
maintainers / contributors) but no persona has a complete path through
the documentation today.

A reader landing on the project today has to read RFCs to understand
what the system *is*. RFCs are excellent design records but poor
introductions. v0.10 closes the gap.

## 2. Persona structure

### 2.1 New user (P1)
Has heard of Fjell, wants to understand whether it fits their problem,
and wants to try it in under an hour.

Required path:
- `docs/src/intro/what-is-fjell.md` — answers "what is this?" in
  one screenful, references RFC 061 §2 (identity statement).
- `docs/src/intro/why-fjell.md` — the three archetypes A1/A2/A3
  from RFC 061 §3.
- `docs/src/intro/non-goals.md` — RFC 061 §3.4 and §7.
- `docs/src/tutorials/quick-start.md` — get to `TEST:M8:PASS` in
  five minutes.
- `docs/src/tutorials/three-node-fleet.md` — the v0.10-005 demo.
- `docs/src/faq.md` — common confusions, including the POSIX question.

### 2.2 Intermediate user (P2)
Has Fjell running, wants to build a service or understand a specific
mechanism.

Required path:
- `docs/src/sdk/overview.md` — `fjell-sdk` tour.
- `docs/src/sdk/writing-a-service.md` — step-by-step.
- `docs/src/sdk/cap-manifest.md` — the CapManifest cookbook.
- `docs/src/sdk/bundle-publishing.md` — the v0.9-004 flow.
- `docs/src/api/syscalls.md` — generated reference (links to
  `cargo doc --no-deps -p fjell-syscall`).
- `docs/src/api/semantic-catalog.md` — generated by the
  `fjell-semantic-toolkit` doc renderer (v0.9-003).
- `docs/src/abi/policy.md` — v0.10-002.
- `docs/src/abi/stability.md` — v0.10-002.

### 2.3 Maintainer / contributor (P3)
Wants to understand a subsystem deeply or contribute back.

Required path:
- `docs/src/architecture/overview.md` — kernel, services, IPC.
- `docs/src/architecture/capability-system.md` — RFC 031, 048, 049.
- `docs/src/architecture/leases.md` — RFC 033, 034.
- `docs/src/architecture/measurement-and-attestation.md` —
  RFC v0.3-001..004.
- `docs/src/architecture/fleet.md` — RFC v0.8-001..005.
- `docs/src/verification/unsafe-charter.md` — already exists; link.
- `docs/src/verification/unsafe-inventory.md` — already exists; link.
- `docs/src/verification/property-tests.md` — RFC v0.6-001 overview.
- `docs/src/identity/v1-direction.md` — full text of RFC 061.
- `docs/src/release/reproducibility.md` — v0.10-003.
- `docs/src/release/v1-readiness.md` — v0.10-007.
- `docs/src/contributing/rfc-process.md` — RFC 000 distilled.

## 3. Truth bond

Every page must satisfy at least one:

- Claim references a passing test in `tests/runs/<recent>/`.
- Claim references a verified invariant from this RFC set or an RFC
  in `done/`.
- Claim is explicitly marked **TODO** with the RFC that will resolve it.

No persona doc may contain unmarked aspirational text. The unsafe
charter sets the precedent.

## 4. README discipline

`README.md` is *not* the documentation. Per the project guidelines it
contains: hero badges, overview, why/when, quick start, design notes,
links to full docs. RFC-v0.10-006 reasserts this and trims the README
to that shape if it has drifted.

A new top-of-README badge links to the latest `summary.json` from
`test-all`, so the README's tests-passing claim is verifiable.

## 5. Build and verify

`cargo xtask docs build` (new) wraps `mdbook build` and adds:

- Link check (no broken internal or external links).
- A lint that fails the build if any TODO has no RFC reference.
- An index check that every persona's required-path file exists.

This is added to the `test-all` host tier.

## 6. Acceptance criteria

1. Every file listed in §2 exists and is non-empty.
2. `mdbook build` succeeds with zero warnings.
3. `cargo xtask docs build` passes the link, TODO, and index checks.
4. The README is at most ~150 lines, follows the structure in
   §4, and has a working test-status badge.
5. A new reader can follow `tutorials/quick-start.md` end-to-end on
   a clean checkout.
6. A new reader can follow `tutorials/three-node-fleet.md`
   end-to-end and produce a Trust Report.

## 7. Out of scope

- Translation / i18n.
- A separate hosted documentation site (mdbook output is enough).
- Video walkthroughs.
- Tutorials for non-targeted workloads (general computing, web,
  desktop).
