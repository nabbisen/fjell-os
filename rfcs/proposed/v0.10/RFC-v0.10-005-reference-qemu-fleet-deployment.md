# RFC-v0.10-005 — Reference QEMU Fleet Deployment

**Status:** Proposed
**Target version:** v0.10.0
**Parent:** RFC 061 (credibility).
**Cross-refs:** RFC v0.8-001..005 (fleet ops), RFC v0.9-004 (bundles),
    RFC-v0.10-004 (benchmarks).

## 1. Problem

Fleet ops shipped in v0.8. The CHANGELOG can demonstrate the *types*
exist, but no one has ever seen Fjell run as a multi-node fleet. The
identity claim in RFC 061 ("verifiable fleet nodes") is only credible
when there is a reproducible deployment that does it end-to-end.

v0.10 ships **one** canonical tutorial deployment in QEMU:

> Three QEMU-emulated RISC-V nodes; one acts as the fleet
> coordinator. A signed bundle is rolled out, measurements flow back,
> a staged rollback is demonstrated.

It is the smallest deployment that exercises every v0.8 capability
honestly.

## 2. Topology

```text
                  ┌──────────────┐
                  │  coordinator │ ← fleetd (M-role coord)
                  │   node-A     │   syncd
                  └──┬─────┬─────┘
                     │     │
            ┌────────┘     └────────┐
       ┌────▼─────┐            ┌────▼─────┐
       │  node-B  │            │  node-C  │
       │  fleet   │            │  fleet   │
       │  member  │            │  member  │
       └──────────┘            └──────────┘
```

All three are `qemu-system-riscv64 -M virt` instances on the host,
connected via QEMU's user-mode networking with port forwarding for
the secure-transportd channel.

## 3. Workflow

The tutorial walks the operator through:

1. Build kernel + services (`cargo xtask build`).
2. Generate a `CapManifest` for an example service `fjell-hello`.
3. Build a `ServiceBundle` for `fjell-hello`.
4. Launch three QEMU instances via `cargo xtask fleet-demo up`.
5. Enrol node-B and node-C with node-A's fleet roster.
6. Stage and roll out the `fjell-hello` bundle from node-A.
7. Observe measurements arriving at node-A.
8. Force a failure on node-C; observe staged rollback.
9. Read the generated Trust Report for the deployment.
10. Tear the deployment down (`cargo xtask fleet-demo down`).

Time budget: end-to-end run on a laptop in under 15 minutes.

## 4. Deliverables

```text
examples/three-node-fleet/
  README.md                 — full tutorial walkthrough
  Cargo.toml                — workspace member for example services
  fjell-hello/              — minimal demo service
    src/main.rs
    cap-manifest.toml
  fleet-demo.toml           — fleet topology definition
  expected/                 — golden Trust Report snippets
    trust-report.txt
    measurements.json
```

A new xtask:

```text
cargo xtask fleet-demo {up|down|status|build-bundle|deploy|verify}
```

`fleet-demo up` launches the three nodes. `fleet-demo verify` checks
the produced Trust Report against `expected/trust-report.txt`.

## 5. CI coverage

A new profile `tests/qemu/profiles/fleet-demo.toml` runs the entire
sequence and looks for:

```text
FLEET:NODE_A:READY
FLEET:NODE_B:ENROLLED
FLEET:NODE_C:ENROLLED
FLEET:BUNDLE_DEPLOYED
FLEET:NODE_B:MEASUREMENT_RECEIVED
FLEET:NODE_C:ROLLBACK_OK
TEST:V0.10-FLEET-DEMO:PASS
```

The xtask `test-all` adds this profile to the QEMU tier.

## 6. Acceptance criteria

1. `examples/three-node-fleet/` exists with the listed structure.
2. The README walks an outside reader through the demo with copy-pasteable
   commands.
3. `cargo xtask fleet-demo up` followed by `verify` succeeds on a
   developer machine in under 15 minutes.
4. The CI profile passes in `test-all`.
5. The Trust Report (RFC 061 §6) generated by the demo contains
   non-zero capability, lease, measurement, and bundle sections.
6. The demo can be torn down and re-run idempotently.

## 7. Out of scope

- Real network (the demo uses QEMU user-mode networking; v0.12 may
  add bridged networking).
- More than three nodes (a stress-scale demo is v0.13 territory).
- Cross-host deployment (single host only).
- Production-quality `fjell-hello` (it is a minimal demonstrator).
- Operator UI / dashboard.

## 8. Why this matters

The reference fleet is the first artefact an outside reader can run
that closes the loop:

```text
identity → capability → lease → bundle → measurement → roster → recovery
```

Without it, every claim in the design documents is hypothetical.
With it, the claim is checkable in 15 minutes.
