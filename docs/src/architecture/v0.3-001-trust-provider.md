# v0.3 RFC-001 — Hardware Trust Provider

**Status:** landed in v0.3.0-alpha.1.

## Position in the stack

```text
verifyd / attestd / measuredd / upgraded
    │
    ▼
HardwareTrustProvider trait        ← provider-neutral surface
    │
    ▼
DevelopmentTrustProvider           ← in v0.3.0-alpha.1
TpmTrustProvider   (future, v0.3.x)
DiceTrustProvider  (future, v0.3.x)
```

The trait is owned by user-space.  The kernel never imports
`fjell-trust-provider`; instead, services request provider operations
through their existing cap-broker-gated IPC surface, and the registry
(below) tracks the active provider set.

## Provider registry

The registry is a fixed-capacity (`MAX_PROVIDERS = 8`) slot table.  Each
slot carries a `TrustProviderDescriptor` (public, semantic-visible) and an
opaque generation counter that bumps on every replace/remove so a stale
`ProviderHandle` always fails the use-time check.

```text
Phase ::= Bootstrap | Enforcing       (one-way)

register(descriptor) -> ProviderHandle
  Bootstrap : assigns next id, generation = 1
  Enforcing : refuses Null providers; otherwise PhaseLocked

replace(handle, descriptor) -> ProviderHandle
  bumps generation; refuses Null in Enforcing
  refuses stale handles with StaleHandle

remove(handle) -> ()
  frees the slot; the old generation never re-occupies it
```

The `enter_enforcing()` transition is monotonic and never reverses; this
matches the v0.2 `cap-broker` Bootstrap→Enforcing pattern.

## Trait surface

`HardwareTrustProvider`'s methods are intentionally narrow:

| Method | Purpose |
|--------|---------|
| `provider_id()` / `descriptor()` | identification |
| `read_measurement()` | current chain head as known to the provider |
| `sign_attestation(digest)` | sign a *pre-domain-separated* digest |
| `read_anti_rollback_counter()` | monotonic counter, persisted by provider |
| `seal_key(purpose, key)` | seal under a `KeyPurpose` |
| `unseal_key(purpose, sealed)` | refuse on `PurposeMismatch` |

Every error variant maps 1:1 to a stable u16 code so the audit subsystem
and semantic stream can project failures losslessly.

## DevelopmentTrustProvider

The development provider is deterministic by construction.  Signatures are
SHA-256 of `TRUST_DOMAIN || provider_id_LE || dev_key || digest`, placed
in the first 32 bytes of the 64-byte signature buffer.  Sealed keys
embed a domain-separated MAC and reject `PurposeMismatch` and any
single-bit corruption with `SealIntegrityFailed`.  This is **not** secure
in any production sense — it is exclusively for QEMU smoke and host unit
testing.

## NullTrustProvider

`NullTrustProvider` returns `NotSupported` for every capability call.
The registry refuses to admit it once `enter_enforcing()` has been
called, so the negative test
`NEG:TRUST:NULL_PROVIDER_FORBIDDEN_IN_RELEASE` has a real object to
point at without ever introducing a security hole.

## Test surface (v0.3.0-alpha.1)

37 host unit tests across five sections: identifier / key-purpose
stability, registry phase semantics, `DevelopmentTrustProvider`
determinism and integrity, the `NullTrustProvider` contract, and the
`Signature` / `KeyMaterial` / `TrustError` type contracts.

See RFC-v0.3-001 §11 for the canonical enumeration.

## Forward-compat notes

- `KeyPurpose::SnapshotSigning = 0x06` is reserved here and consumed by
  v0.7 RFC 002 (`SignedSnapshot`).
- `KeyPurpose::BoardProfile = 0x07` is reserved by v0.5 RFC 001.
- `KeyPurpose::FleetRoot = 0x08` is reserved by v0.8 RFC 001.
- `KeyPurpose::ServiceSigning = 0x09` is reserved by v0.9 RFC 004.

The trait will not change shape for any of these; only new purposes are
added.
