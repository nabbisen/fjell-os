# v0.3 RFC-002 — Keyring, Key-Purpose, Signature-Provider

**Status:** landed in v0.3.0-alpha.1.

## What the keyring is

A fixed-size table of `TrustAnchor` rows, indexed by `KeyPurpose`.  Each
purpose holds up to `ANCHORS_PER_PURPOSE = 4` anchors, ordered by
monotonic `KeyEpoch`.

```text
KeyPurpose ::= ReleaseVerification   (0x01)
             | RootfsVerification    (0x02)
             | PolicyVerification    (0x03)
             | AttestationSigning    (0x04)
             | SealedDataKey         (0x05)
             | SnapshotSigning       (0x06)
```

(`BoardProfile = 0x07`, `FleetRoot = 0x08`, and `ServiceSigning = 0x09`
are reserved by v0.5 / v0.8 / v0.9 RFCs respectively but are not yet
slotted in the v0.3.0-alpha.1 container.)

## Anchor shape

```text
TrustAnchor {
    purpose:    KeyPurpose,
    algorithm:  SignatureAlgorithm,  // Ed25519 | EcdsaP256 | DevDigest32
    authority:  AuthorityClass,      // Genesis | Standard
    epoch:      KeyEpoch,            // monotonic u32
    key_len:    u8,
    key_bytes:  [u8; 64],
}
```

`Genesis` anchors are baked into the OS image; `Standard` anchors arrive
via rotation envelopes or `KeyringSnapshot` imports.  The container does
not distinguish them at lookup time — both serve as verification
authorities — but the audit subsystem projects the class so operators see
where trust originates.

## Install rules

```text
install(anchor):
    if release_mode and not anchor.algorithm.permitted_in_release():
        Err(ReleaseModeViolation)
    if anchor.epoch <= max(existing anchors' epoch for this purpose):
        Err(EpochRegression)
    if no free slot: evict the lowest-epoch existing entry
    place anchor in slot
    Ok(())
```

Epoch regression is the keyring's only "softer than monotonic" rule.  It
specifically rejects equal epochs, not only lower ones, so a rotation
PR cannot accidentally land twice.

## Release mode

`enter_release_mode()` is one-way (mirroring the trust-provider
registry's `enter_enforcing()`).  After it returns, `DevDigest32`
algorithms are refused.  Combined with the trust-provider registry's
Null-rejection, this gives v0.3 a clean dev/release boundary that can be
flipped atomically by `verifyd` at the end of bootstrap.

## DevSignatureProvider

```text
sig = SHA256( KEYRING_DOMAIN || "DEV-V1" || alg_tag || epoch_LE
              || anchor.key_bytes || payload )
```

The first 32 bytes of the 64-byte signature buffer hold the digest; the
rest is zero.  Verification is a constant-time compare.  Both sign and
verify require `anchor.algorithm == DevDigest32`.

`DevSignatureProvider` is the only path that lets `attestd` produce
testable signatures during development.  Production providers will land
under v0.3.x and target Ed25519 / ECDSA P-256.

## KeyringSnapshot

Snapshot format v1:

```text
KeyringSnapshot {
    schema_version : u16,
    anchor_count   : u8,
    anchors        : [Option<TrustAnchor>; MAX_SNAPSHOT_ANCHORS],
    snapshot_digest: Digest32,
}
```

Canonical digest domain: `FJELL-KEYRING-V1 || SNAP-V1`.  Each anchor is
serialised in fixed order — purpose, algorithm, authority, epoch,
key_len, key_bytes — preceded by a present/absent flag.  `apply_to(&mut
Keyring)` recomputes the digest, rejects mismatches with
`SnapshotDigestMismatch`, and installs anchors back through the normal
`install()` path so all invariants (epoch monotonicity, release-mode
restriction) re-apply on import.

## Test surface (v0.3.0-alpha.1)

29 host unit tests across five sections: algorithm / anchor / epoch
contracts, container install/lookup/epoch rules, `DevSignatureProvider`
round-trip and rejection paths, `KeyringSnapshot` persistence, and
`SigError` code stability.
