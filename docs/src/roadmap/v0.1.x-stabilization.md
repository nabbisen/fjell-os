# Fjell OS v0.1.x — Stabilisation Roadmap

**Status:** Active.
**Defined by:** RFC 024 (also known as RFC-v0.1.x-001).
**Source documents:**
[`fjell-os-v0.1.x-rfcs.md`](https://github.com/nabbisen/fjell-os/blob/main/docs/plans)
and the corresponding RFCs **024–047** in this repository.

This document is the **plan of record** for the v0.1.x release
line. v0.1.x stabilises the v0.1.0 prototype and prepares it for
v0.2.0 (*Security Boundary Closure*).

---

## Purpose

v0.1.x has one goal:

> Make v0.1.0 testable, documented, and auditable so that v0.2.0
> can change security boundaries without regressing.

It does **not** add OS functionality. Any RFC that would add
functionality is rejected for v0.1.x and either deferred to v0.2
or filed against a later milestone.

## Release sequence

| Version | Theme | RFCs landed |
|---|---|---|
| v0.1.1 | Release freeze + CI foundation | 024, 025 |
| v0.1.2 | Negative test harness + threat model + ABI inventory | 026, 027, 028 |
| v0.1.3 | Capability/Lease, MMIO/DMA, evidence audits | 029, 030, 044 |
| v0.1.4 | ADR sync + release checklist | 045, 046 |
| v0.1.5 | v0.2 preparation backlog | 047 |

After v0.1.5 the project enters v0.2.

## v0.1.1 — Release freeze + CI foundation (this release)

### Done

- RFC 024 — release freeze and scope declaration.
- RFC 025 — CI / QEMU automation foundation.
- `v0.1.0-scope.md`, `v0.1.0-limitations.md`,
  `v0.1.0-known-non-goals.md`, `v0.1.0-threat-model.md`
  (skeleton).
- `fjell-tools` extended with `qemu-negative`, `qemu-log-check`,
  `qemu-run --profile`.
- `.github/workflows/ci.yml` with five jobs (format, check, host
  test, qemu-smoke, qemu-negative).
- README links to the limitations document.
- ROADMAP declares v0.2 as Security Boundary Closure.

### Deferred from v0.1.1 to v0.1.2

- Full threat model body (RFC 027).
- Negative-test cases themselves (RFC 026 lands the
  *infrastructure*; case bodies land per v0.2 RFC).

Targets RFC 047. Output:

- Every v0.2 RFC has at least one backlog entry pointing to it.

## Exit criteria

v0.1.x is complete and v0.2 may begin when:

- CI runs host tests and QEMU smoke tests on every PR.
- A negative test harness exists and runs in CI.
- v0.1.0 limitations are documented (see this directory + `security/`).
- A threat model exists (`security/threat-model-v0.1.md`).
- An ABI inventory exists (`abi/v0.1-inventory.md`).
- Capability / Lease enforcement audit exists.
- MMIO / DMA boundary audit exists.
- Evidence export audit exists.
- ADRs are synchronised.
- A release checklist exists.
- A v0.2 backlog is ready.

At that point the v0.2 RFC set (031–043) is unblocked.
