# Fjell OS v0.1.0 — Limitations

**Status:** Frozen at v0.1.0 (May 2026).
**Defined by:** RFC 024 (filename `024-release-freeze-and-scope-declaration.md`,
also known as RFC-v0.1.x-001).

This document declares **what v0.1.0 is not**. It exists so that
contributors and downstream users cannot mistake an early local
prototype for a hardened production operating system.

If your use case requires anything below, **do not deploy
v0.1.0**. Wait for the milestone that addresses it (each section
links to the milestone) or evaluate a different operating system.

---

## v0.1.0 is explicitly not

The following statements are normative:

- **not production secure boot**
- **not hardware-rooted trust**
- **not remote attestation**
- **not networked update**
- **not general-purpose networking**
- **not POSIX**
- **not a desktop OS**
- **not a multi-user production OS**
- **not fully verified**
- **not yet security-boundary-complete**

## What this means in practice

### "not production secure boot"

The boot path verifies signed releases using a development-grade
Ed25519 stand-in implemented via SHA-256 keyed under
`dev-attest-m8-01`. The key is in-tree. This is sufficient for
local smoke testing and for verifying the data-flow shape, but it
is **not** cryptographically meaningful against an attacker who
can read the source tree (i.e. anyone). Production secure boot
requires hardware-rooted trust (TPM / DICE / PUF), which is a
v0.3.0 milestone (*Hardware Trust Abstraction*).

### "not hardware-rooted trust"

There is no TPM integration, no DICE measurement chain rooted in
firmware, no fused identity, no manufacturer key. The measurement
chain runs entirely in software inside the running kernel and
records what that kernel chose to record. A compromised kernel can
forge any measurement value.

### "not remote attestation"

The local attestation record exists, but there is no transport
that delivers it to a remote verifier, no nonce protocol against
replay, no challenge/response. v0.4.0 (*Minimal Secure Networking*)
is the earliest version that could add this; remote attestation
itself is later still.

### "not networked update"

Updates are delivered as files placed into the QEMU disk image.
There is no in-system download path. There is no signed-mirror
protocol. There is no rollback notification to a remote console.

### "not general-purpose networking"

There is no TCP stack, no UDP stack, no socket API, no DNS, no
virtio-net driver. Adding networking would be premature before
security closure (v0.2.0). The first version that introduces
networking is v0.4.0, and *only* for attestation, diagnostics,
and update metadata — not for general application use.

### "not POSIX"

No `open` / `read` / `write` / `fork` / `exec`. No shell. No
filesystem in the POSIX sense. Applications interact with the
system through Fjell's own syscall set (see the upcoming
`docs/abi/v0.1-inventory.md` produced by RFC 028 in v0.1.2).

### "not a desktop OS"

No window system. No keyboard or mouse. No framebuffer driver
shipping in tree. The UI boundary is the *Intent Stream* — services
emit semantic intent and a renderer (today `fjell-proxy-text`)
projects it into a modality. Adding a graphical renderer is not on
the v0.x roadmap.

### "not a multi-user production OS"

There is no user account model. There is no authentication. The
capability system is the authority model, but the v0.1.0 policy
shipped with the system trusts every spawned task implicitly. v0.2
adds default-deny (RFC 040).

### "not fully verified"

There are no formal proofs of correctness. There is no Coq, no
Isabelle, no F*, no SAW. Property-based testing exists in places
(M8 acceptance criteria), but full verification is a v0.6
milestone.

### "not yet security-boundary-complete"

The most important limitation: the boundaries between kernel and
services, services and drivers, drivers and devices, services and
persistent storage — **exist as types and as data**, but their
enforcement is not yet uniform. v0.2.0 is the closure milestone.

## The path forward

| Limitation | Closed in |
|---|---|
| security boundary enforcement | v0.2.0 (RFCs 031–043) |
| hardware-rooted trust | v0.3.0 |
| minimal networking | v0.4.0 |
| multi-platform support | v0.5.0 |
| verification gates | v0.6.0 |

For the full plan see `docs/src/roadmap/v0.1.x-stabilization.md`
and the project-level v0.1-to-v0.9 roadmap.
