# Fjell OS v0.2.0 — Security Boundary Closure Release Gate

> RFC 043 — v0.2 Security Boundary Release Gate  
> Gate token: **`TEST:V02:PASS`**

## Summary

v0.2.0 is the "Security Boundary Closure" release.  It closes the open gaps
identified by the v0.1.0 enforcement audits (RFC 029, RFC 030, RFC 044) and
delivers a fully-wired capability enforcement stack from the kernel through
the capability broker to user space.

## Known limitations in v0.2.0

These are explicitly accepted limitations, tracked for v0.3 or later:

1. **Task/lease syscalls use slot-scan** (V02-A-001): `sys_task_spawn`,
   `sys_task_start`, etc. still use CSpace slot-scan for capability checking.
   Handle-based `require_cap` at each call site requires ABI changes.  Tracked
   for v0.3.

2. **DMA quarantine timeout is synchronous zeroize**: RFC 036 quarantine
   timeout requires timer-callback integration for uncertain device-quiesce.
   Deferred — no use case at v0.2 scale.

3. **Service extraction incomplete**: storaged, bootctl, and other services
   remain inline in fjell-init per ADR-0010.  RFC 038 infrastructure
   (READY protocol types, service-manager lifecycle) is defined; binary
   extraction requires QEMU testing.

4. **Badge-based sender identity in cap-broker bootstrap**: The
   `BOOTSTRAP_COMPLETE` handler trusts the kernel's delivery without verifying
   `sender_badge == ImageId(init)`.  Requires kernel badge enforcement.

5. **Owner-scope MMIO enforcement**: MmioRegion caps are validated for kind and
   rights but not for "this cap belongs to this specific task".  Cap-broker
   policy-layer enforcement (RFC 040) covers this at grant time.
