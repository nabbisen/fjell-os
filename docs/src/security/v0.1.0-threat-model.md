# Fjell OS v0.1.0 — Threat Model (Skeleton)

**Status:** Skeleton (v0.1.1). Expanded by RFC 027 in v0.1.2.
**Defined by:** RFC 024 §"Required Documents" mandates this file
exist by v0.1.1; RFC 027 (filename

This file is the placeholder so README and SUMMARY links resolve.
The full threat model arrives in v0.1.2.

---

## 1. Scope

Fjell OS v0.1.0 running on QEMU `virt`, single hart, no network.
The threat model assumes the source tree is public and an attacker
can read every line.

## 2. Assets

- kernel memory
- task address spaces
- capability tables
- lease table
- endpoint table
- DMA regions
- persistent state store
- boot-control block
- signed release metadata
- signed policy metadata
- immutable rootfs metadata
- snapshot records
- measurement chain
- local attestation records

## 3. Trusted Computing Base (v0.1.0)

- kernel binary
- service binaries embedded by `build.rs`
- the development-grade signing key (`dev-attest-m8-01`)

The TCB is **maximal** in v0.1.0: every embedded binary is part of
the TCB. v0.2 work narrows this.

## 4. Attacker Model

In scope:

- a hostile or buggy *task* trying to invoke privileged syscalls.
- a hostile or buggy *service* trying to invoke unauthorised IPC.

Out of scope (explicitly):

- physical attacker
- hardware DMA attacker
- malicious firmware
- compromised boot ROM
- remote attacker over network
- supply-chain compromise beyond development-grade signing
- side-channel attacks
- SMP race attacks

## 5. Trust Boundaries

Each boundary listed here is *defined* in v0.1.0 but not all are
fully *enforced*. RFC 027 documents the enforcement state of each.

- Kernel ↔ Task
- Service ↔ Service (via IPC + capability)
- Service ↔ Driver (via MMIO / DMA capabilities)
- Persistent Store ↔ Service (via storaged)
- Verified Artefact ↔ Recovery (via verifyd / recoveryd)

## 12. Known Weaknesses

The full enumeration is owned by the v0.1.3 audit RFCs (029, 030,
044). For v0.1.1, the most important known weaknesses are:

- `caller_has_cap(kind)` style checks remain in some syscall paths.
- `sys_mmio_map` accepts caller-supplied physical addresses.
- DMA cleanup is not deterministic on all paths.
- `cap-broker` does not yet enforce default-deny.

## 13. Deferred Threats

The threats listed under §4 *out of scope* are deferred. The
earliest version that addresses each:

- physical attacker, hardware DMA, malicious firmware: v0.3.0
- remote attacker: v0.4.0 (and only for the introduced surface)
- SMP races: when SMP is introduced (no current milestone)
- side-channel: no current milestone

## 14. v0.2 Security Boundary Closure Plan

See the v0.2 RFC set (RFCs 031–043) and the v0.2 overview design
document. v0.2 closes:

- Unified capability enforcement (RFC 031).
- Lease epoch revocation (RFC 033) with blocked IPC wake/cancel
  (RFC 034).
- MmioRegion capability ABI replacement (RFC 035).
- DmaRegion capability + zeroize/quarantine (RFC 036).
- Cooperative service separation with timer fail-safe (RFCs 037,
  038).
- Safe user copy and real audit drain (RFC 039).
- cap-broker default-deny with explicit bootstrap (RFC 040).
- Persistent evidence hardening (RFC 041).
- Expanded negative-test matrix and binding release gate (RFCs
  042, 043).
