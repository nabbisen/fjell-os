# ADR-0005 — v0.1.0 Scope

**Status:** Superseded by RFC 024  
**Date:** 2026-05-04

## Context

What does v0.1.0 include, and — equally important — what does it deliberately
exclude?

## Decision

### In scope

1. Bootable RISC-V 64 kernel on QEMU `virt` (M1)
2. Memory isolation: Sv39, frame allocator, user address spaces (M2)
3. Task scheduling and user-mode execution (M2)
4. Trap, syscall, and user-fault containment (M2)
5. Synchronous IPC and capability table (M3)
6. `fjell-init`, `fjell-service-manager` (M4)
7. Append-only audit ring + `fjell-auditd` (M5)
8. TOML config validation + `fjell-configd` (M6)
9. Intent Stream schema + `fjell-proxy-text` (M7)
10. Hardening, property tests, unsafe review, release tag (M8)

### Out of scope for v0.1.0

- SMP / multi-hart scheduling
- Demand paging, copy-on-write, `mmap`
- Persistent filesystem (beyond in-memory append-only ring)
- Network stack
- Virtio block/net drivers (beyond what M4 needs for static service images)
- Dynamic ELF loading
- Linux / POSIX compatibility
- GUI stack
- AI inference integration
- Formal verification (infrastructure for it is laid, not executed)
- Real hardware support beyond QEMU

## Rationale

v0.1.0 is a proof-of-concept that demonstrates the core thesis:

> A memory-safe minimal kernel + capability IPC + user-space services +
> append-only audit + semantic stream can form a coherent, small,
> extensible OS.

Shipping that proof-of-concept as a clean, well-documented, testable base is
more valuable than shipping half-baked drivers or a half-proven IPC model.

## Consequences

- Each milestone is gated by a concrete acceptance criterion and a QEMU
  smoke test.
- M2 is the first "hard gate": nothing in M3+ makes sense without proven
  memory isolation and fault containment.
- Scope discipline is enforced at PR review time; any feature not listed
  above is deferred without negotiation.
