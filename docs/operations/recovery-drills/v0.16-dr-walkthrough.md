# Recovery Drill Attestation — DR Walkthrough v0.16

**Drill date:** v0.16.0 validation closure
**Driver:** validation closure sprint (non-author of the recovery guide)
**Target:** QEMU `virt` reference fleet (host-simulated where noted)
**RFC:** RFC-v0.16-003 (closes errata E-005, E-008)

This attests that the recovery procedures in
`docs/operations/recovery-guide.md` were walked through against the
reference environment, not merely written. Each scenario below records
the procedure followed, the observed result, and any deviation from
the documented steps.

---

## DR1 — Coordinator loss (recovery-guide §3.4)

**Procedure followed:**
1. Simulated coordinator node halt (removed from fleet membership).
2. Surviving members entered `Partitioned, no coordinator` per the FSM —
   confirmed via `fjell-fleet-sync` drill
   (`DRILL:FLEET-PARTITION-RECONCILE:PASS`).
3. Operator promotion: constructed a `CoordinatorPromotion` record
   signed by the `TrustAnchorRoot`.
4. Promoted member became coordinator; remaining members re-attached.

**Observed:** the FSM refused any self-promotion path; promotion required
the explicit operator-signed record, matching RFC-v0.13-005's
"operator promotion is the only path" guarantee.

**Deviation:** none. The guide's claim that an offline operator equals an
offline fleet held exactly — this is a documented limitation, not a bug.

## DR2 — Key compromise (recovery-guide §3.5)

**Procedure followed:**
1. Generated a fresh signing key (now encrypted at rest, FJK2, per
   RFC-v0.16-006).
2. Built a `RevocationRecord` for the compromised key
   (reason = `Compromised`).
3. Advanced the `RevocationTable`: Active → Revoked (terminal).
4. Re-signed the current bundle under the new key.
5. Verified the new bundle with `verify-bundle-sig` → PASS.
6. Confirmed a bundle signed by the revoked key would be refused once
   the revocation propagates (table state = Revoked).

**Observed:** the revocation FSM rejected a second revocation of the same
key (idempotent) and refused Active→Active. Re-signing and verification
round-tripped cleanly with the encrypted key.

**Deviation:** the guide referenced "passphrase-protected key" — this is
now true (E-002 closed). Prior to v0.16 the key was plaintext; the
drill confirms the encrypted path works end to end.

## DR5 — Local audit storage corruption (recovery-guide §3.7)

**Procedure followed:**
1. Simulated a corrupted summary (regressed `sync_seq`).
2. Ran `fjell-summary-check` against the corrupted summary.
3. Checker flagged `SyncSeqRegression`.

**Observed:** `DRILL:FLEET-PARTITION-ROLLBACK-REJECTED:PASS` — the
consistency checker caught the regression, matching the guide's
expectation that a rolled-back node is detected at rejoin.

**Deviation:** none.

Covered by the integration drill in

full partition → divergent writes → reconnect → reconcile manifest →
apply → consistency check. Marker:
`DRILL:FLEET-PARTITION-RECONCILE:PASS`.

## Boot failure triage (recovery-guide §3.1)

**Procedure followed:**
1. Fed `fjell-dtb-validate` a DTB missing the UART node.
2. Validator returned `R4RequiredDeviceMissing`.
3. Followed the guide's "DTB mismatch → update board-profile.toml" step.

**Observed:** the failure code matched the guide's documented symptom
table (`FJELL-BOOT-FAIL: DTB R4 <addr>`). Triage step resolved it.

**Deviation:** none.

## Attestation

All walked scenarios produced the documented outcomes. Two markers were
captured in automated form (`PARTITION-RECONCILE`, `ROLLBACK-REJECTED`);
the remainder were exercised against the real crate APIs
(`RevocationTable`, `CoordinatorPromotion`, `fjell-dtb-validate`,
`fjell-summary-check`) rather than only read.

The recovery guide is hereby attested as **walked, not just written**,
for scenarios DR1, DR2, DR5, partition handling, and boot triage.

Scenarios DR3, DR4, DR6, DR7, DR8 (full disaster-recovery set) remain
**document-only** at v0.16 and are not claimed as drilled; they are
listed in the v1.0 limitations section (RFC-v0.16-005).
