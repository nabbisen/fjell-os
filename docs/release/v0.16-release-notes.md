# Fjell OS v0.16.0 Release Notes

**Theme:** Validation Closure Sprint
**Type:** Pre-release — gate evidence for the v1.0.0 tag decision
**Date:** 2026-05-28

---

## Purpose of this release

v0.16 converts the paper claims of v0.10–v0.15 into validated ones.
No new architecture was introduced. All work was directed at resolving
the five pre-v1.0 blockers and five high-priority items identified in
the architect's review of the v0.9–v0.15 handoff.

---

## What changed

### RB-01 Ed25519 sign-path conformance (RFC-v0.16-001)

The RFC 8032 §7.1 TV1 "discrepancy" was traced to a transcription error
in the test-vector seed constant (corruption from byte 15 onward — the
crypto library was always correct). Cross-verified against three
independent implementations (ed25519-dalek, OpenSSL, libsodium): all
three agree on the canonical triple. Both removed TV1 tests restored and
passing. The Fjell signing path now has independently-evidenced RFC 8032
conformance.

### RB-03 Fleet partition drill (RFC-v0.16-002)

A full-lifecycle integration drill exercises the `fjell-fleet-sync`
runtime path end-to-end: Healthy → Suspect → Partitioned → divergent
writes → Reconciling → `ReconcileManifest` (accept coordinator records,
reject conflicting partition-side records) → apply → Healthy → summary
consistency check. A second arm confirms rollback detection.

Markers: `DRILL:FLEET-PARTITION-RECONCILE:PASS`,
`DRILL:FLEET-PARTITION-ROLLBACK-REJECTED:PASS`.

### RB-04 Recovery drill and attestation (RFC-v0.16-003)

Recovery procedures in `docs/operations/recovery-guide.md` were walked
against real crate APIs (DR1 coordinator loss, DR2 key compromise,
DR5 audit corruption, partition handling, boot triage). Attestation
committed to `docs/operations/recovery-drills/v0.16-dr-walkthrough.md`.

### RB-05 RFC errata governance (RFC-v0.16-004)

New status `Implemented-with-Errata` added to the lifecycle policy.
New file `docs/rfcs/ERRATA.md` records E-001 … E-009. At v0.16 close:
0 OPEN, 8 CLOSED, 1 ACCEPTED. Seven prior RFCs re-marked accordingly.

Standing rule added: cryptographic test vectors must be cross-verified
against an independent implementation at authoring time.

### H-01 Encrypted key storage (RFC-v0.16-006)

Signing keys now encrypted at rest — `FJK2` format: Argon2id key
derivation + AES-256-GCM encryption of the 32-byte seed. Passphrase
from `--passphrase` or `FJELL_KEY_PASSPHRASE` env. Public key stored
cleartext at a fixed offset so `key show` needs no passphrase. Wrong
passphrase rejected; tampered ciphertext detected (GCM tag). Legacy
plaintext `FJKY` retained only behind `--insecure-plaintext` for CI.

### H-04 Repro-check hardened to SHA-256

The reproducible-build gate digest switched from FNV-1a to SHA-256.
The gate's purpose remains change detection; the switch removes the
external confusion of a non-cryptographic digest in trust tooling.

### H-05 Runtime SDK trial (RFC-v0.16-007)

`fjell-config-sync` driven through a full update lifecycle via its real
handler entry points: cold start, update, idempotent re-apply, query,
semantic emit eligibility, unknown-tag rejection. A convergence arm
confirms two independent instances agree on the digest for the same blob.

Markers: `DRILL:SDK-CONFIG-SYNC-RUNTIME:PASS`,
`DRILL:SDK-CONFIG-SYNC-CONVERGENCE:PASS`.

### Release rehearsal gate (RFC-v0.16-008)

`cargo xtask release-rehearsal` runs all 8 mechanical v1.0 tag gates and
prints a PASS/FAIL matrix. All 8 gates pass in this release.

## Limitations — what v1.0 does and does not claim

These limitations define the scope of the v1.0 tag when it is approved.
v1.0 is the first supported **QEMU `virt` RISC-V profile**; it is not a
validated hardware release.

### v1.0 does NOT claim

| Item | Detail |
|------|--------|
| Validated real-hardware deployment | VisionFive 2 profile is provisional; no hardware was booted (errata E-004, ACCEPTED) |
| Production industrial-gateway readiness | QEMU developer profile only |
| Multi-VM / wire-level fleet validation | Partition drill is host-simulated; wire transport unproven |
| Full disaster-recovery rehearsal | DR1/DR2/DR5 + partition + boot drilled; DR3/DR4/DR6/DR7/DR8 document-only |
| Hardware-rooted trust | `TrustAnchorRoot` provisioning undefined; tracked as RFC-v0.17-001 |
| Multi-hart safety | Single-hart (hart 0) only; MMIO audit is single-hart scoped |
| POSIX compatibility | No POSIX surface |
| Kernel-mediated IPC for SDK reference service | Runtime trial is handler-level, not kernel-switchboard |
| Byte-level `ZeroizeOnDrop` guarantee | In-memory seed erase-on-drop not independently verified |
| Complete semantic ABI checking | ABI gate is a drift guard; does not catch all semantic breakage |

### v1.0 DOES claim

- QEMU `virt` RISC-V boots the kernel (M1–M8 smoke markers earned).
- Capability-based, lease-bounded authority (RFC 061 §4 invariants preserved).
- Ed25519 bundle signing: RFC 8032 conformant, cross-verified against
  three independent implementations.
- Replay-attack resistance via nonce table and sliding-window cache.
- Key revocation with signed `RevocationRecord` (FSM: Active → Retired/Revoked).
- Fleet-sync types and reconcile path exercised end-to-end (host-simulated).
- SDK surface runtime-sufficient for a stateful reference service.
- Recovery procedures documented and walked for the drilled scenarios.
- All gate tooling automated via `cargo xtask release-rehearsal`.
