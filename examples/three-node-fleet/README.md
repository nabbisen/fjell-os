# Fjell OS — Three-Node QEMU Fleet Demo

*Implements RFC-v0.10-005. Demonstrates the complete Fjell fleet flow:
build → enrol → deploy bundle → measure → verify Trust Report.*

**Time budget:** ≤ 15 minutes on a laptop with QEMU installed.

---

## Prerequisites

- Rust toolchain pinned in `rust-toolchain.toml`
- `qemu-system-riscv64` ≥ 7.0 on `PATH`
- 500 MB free disk

Verify:
```bash
qemu-system-riscv64 --version | head -1
# QEMU emulator version 7.x.x
cargo --version
```

---

## Step 1 — Build the kernel

```bash
cargo xtask build
# Produces: target/riscv64gc-unknown-none-elf/release/fjell-kernel
```

---

## Step 2 — Validate the demo service manifest

```bash
cargo xtask dev lint examples/three-node-fleet/fjell-hello/cap-manifest.toml
# dev lint: OK  service=`fjell-hello` sdk_api_rev=1
```

---

## Step 3 — Build and validate the bundle

```bash
cargo xtask fleet-demo build-bundle
# [fleet-demo] manifest lint: PASS
# [fleet-demo] bundle-build PASS
```

---

## Step 4 — Launch the three QEMU nodes

```bash
cargo xtask fleet-demo up
# [fleet-demo] launched node-a (coordinator) on serial port 4440
# [fleet-demo] launched node-b (member)      on serial port 4441
# [fleet-demo] launched node-c (member)      on serial port 4442
```

Each node's serial output is captured to `tests/fleet-demo/logs/<node>.log`.

---

## Step 5 — Observe boot

```bash
# In another terminal, follow node-a's serial log:
tail -f tests/fleet-demo/logs/node-a.log
# Expected within 30s:
#   kernel: boot
#   service-manager: all services ready
#   FLEET:NODE_A:READY
```

---

## Step 6 — Verify all pass markers

```bash
cargo xtask fleet-demo verify
# [fleet-demo] waiting for pass markers (120s timeout) …
# [fleet-demo] PASS — all markers observed
```

Pass markers required:
- `FLEET:NODE_A:READY`
- `FLEET:NODE_B:READY`
- `FLEET:NODE_C:READY`
- `TEST:V0.10-FLEET-DEMO:PASS`

---

## Step 7 — Generate the Trust Report

```bash
cargo xtask trust-report
# [trust-report] written to docs/release/trust-report.txt
```

Open `docs/release/trust-report.txt` — six sections covering capability
inventory, lease inventory, measurement chain, catalog binding, unsafe
inventory, and CI evidence.

---

## Step 8 — Tear down

```bash
cargo xtask fleet-demo down
# [fleet-demo] sent SIGKILL to 3 QEMU process(es)
```

---

## One-command deploy (steps 2–6 in sequence)

```bash
cargo xtask fleet-demo deploy
# Runs: build-bundle → up → (10s boot wait) → verify → down
```

---

## Demonstrating a rollback

The following scenario exercises RFC-v0.9-004 §5.3 (RolledBack lifecycle):

```bash
# After `cargo xtask fleet-demo up`:
# 1. Observe node-c enter RolledBack state by injecting a
#    BUNDLE_HEALTH:FAILED marker (see docs/dev/modes.md for --trace).
# 2. The coordinator logs BUNDLE.ROLLBACK_TRIGGERED.
# 3. `cargo xtask fleet-demo verify` still passes (rollback is confirmed).
```

Full rollback testing lands in v0.13 (RFC-v0.13-004).

## Common failures

| Symptom | Cause | Resolution |
|---------|-------|------------|
| `kernel not found` | Build step skipped | Run `cargo xtask build` |
| `verify` times out | QEMU not on PATH | Install `qemu-system-riscv64` |
| `verify` times out | Ports in use | Run `cargo xtask fleet-demo down` first |
| `manifest lint: FAIL` | cap-manifest changed | Run `cargo xtask dev lint` |
